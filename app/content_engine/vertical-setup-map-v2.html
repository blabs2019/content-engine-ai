Vertical Auto-Discovery

VerticalRegistryautomatic

Pipeline receives a request with industry: "restaurant". The VerticalRegistry fetches /api/v1/verticals (cached 10min), fuzzy-matches, returns the ID. No hardcoded maps.

Resolution chain business_context.industry = "restaurant"
  → VerticalRegistry.resolve("restaurant")
  → exact match in API list → {name: "restaurant", id: 3}

Fuzzy examples:
  "HVAC" → alias → "plumbing" → id=1
  "law firm" → alias → "legal" → id=4
  "italian bistro" → fuzzy → "restaurant" → id=3
vertical_registry.py — singleton, thread-safe, cached
↓ vertical ID resolved → fetch trending data

Trending Data + Style References

trending_integration.pyautomatic

Fetches viral posts, news, hashtags using auto-resolved ID. Viral posts become style references. No DB table needed.

What flows into the pipeline social_pulse: 20 viral posts → Stage 2 style references
topics: 15 trending topics → Stage 1 topic ideas
news: 10 industry articles → Stage 1 timely triggers
hashtags: top 10 → Stage 2 hashtag guidance
Falls back gracefully if trending data is sparse

Auto-Generate Learnings

reference_resolver.pybackground thread

First request: compose-brief returns empty. Resolver fires a background thread to analyze trending data and generate learnings. Auto-approved at medium confidence. Current request uses fallbacks; next request gets real learnings.

Background flow (non-blocking) 1. compose-brief(vertical="restaurant") → no learnings
2. _trigger_auto_learnings("restaurant") → background thread
3. POST /api/analyze-from-trending → AI analyzes viral posts
4. POST /api/learnings/bulk-approve {confidence: "medium"}
5. Next request → learnings available!

Only triggers once per vertical per process lifetime.
Non-blocking — doesn't slow the request Humans can promote to high confidence later
